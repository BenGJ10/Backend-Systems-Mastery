# Introduction to Git Rebasing

## Overview

**Git rebase** is a command used to **integrate changes from one branch onto another by rewriting commit history**.

Instead of creating a merge commit, rebasing **moves or reapplies commits** from one branch onto a new base commit.

Rebasing is commonly used to maintain a **clean, linear project history** by eliminating unnecessary merge commits.

---

## Why Rebasing Exists

In collaborative development, multiple developers work on different branches simultaneously.
Over time, the base branch (usually `main` or `develop`) continues to receive new commits.

Rebasing allows a feature branch to **replay its commits on top of the latest version of the base branch**, ensuring it incorporates the newest changes while maintaining a linear history.

---

## Basic Example

Assume the following commit history:

```
main:
A --- B --- C

feature:
         \
          D --- E
```

The feature branch was created from commit `B`.
Meanwhile, `main` progressed to commit `C`.

Using rebase:

```bash
git checkout feature
git rebase main
```

Result:

```
A --- B --- C --- D' --- E'
```

Commits `D` and `E` are **reapplied on top of `C`**, creating new commits `D'` and `E'`.

---

## How Rebase Works Internally

Rebase performs the following steps:

1. Temporarily removes commits from the current branch
2. Moves the branch pointer to the target base
3. Reapplies each commit sequentially

Conceptually:

```
Original commits: D → E
Base moved to: C
Reapplied commits: D' → E'
```

Because commits are recreated, their **hash values change**.

---

## Basic Rebase Command

Rebase the current branch onto another branch:

```bash
git rebase <target-branch>
```

Example:

```bash
git rebase main
```

This replays the current branch's commits on top of `main`.

---

## Rebase Workflow Example

Typical feature development workflow:

1. Create a feature branch

```bash
git checkout -b feature-login
```

2. Work and create commits

```bash
git add .
git commit -m "implement login logic"
```

3. Update the branch with latest changes from `main`

```bash
git fetch origin
git rebase origin/main
```

4. Resolve conflicts if they occur, then continue.

---

## Handling Conflicts During Rebase

Conflicts may occur when rebasing if both branches modified the same code.

Example output:

```
CONFLICT (content): Merge conflict in AuthService.java
```

Resolve conflicts by editing the file and then run:

```bash
git add <file>
git rebase --continue
```

To cancel the process:

```bash
git rebase --abort
```

---

## Merge vs Rebase

Both commands integrate changes from another branch, but they behave differently.

| Aspect           | Merge                       | Rebase                              |
| ---------------- | --------------------------- | ----------------------------------- |
| History          | Preserves branching history | Creates linear history              |
| Commit Structure | Adds merge commits          | Rewrites commits                    |
| Safety           | Safe for shared branches    | Dangerous if history already shared |
| Usage            | Default team workflow       | Local branch cleanup                |

---

## Visual Comparison

### Merge

```
A --- B --- C (main)
      \ 
       D --- E (feature)

git merge main
```

Result:

```
A --- B --- C
      \      \
       D --- E --- M
```

`M` is a merge commit.

---

### Rebase

```
A --- B --- C (main)
      \
       D --- E (feature)
```

After:

```
A --- B --- C --- D' --- E'
```

No merge commit is created.

---

## When to Use Rebase

Use rebase when:

* Updating a feature branch with the latest base branch changes
* Cleaning up local commit history before pushing
* Maintaining a linear project history

---

## When Not to Use Rebase

Avoid rebasing when:

* Commits have already been pushed to a shared branch
* Other developers depend on the existing commit history
* Working on public branches like `main`

Rewriting shared history can cause synchronization problems.

---

## Interactive Rebase (Brief Overview)

Git also supports **interactive rebasing**, which allows developers to modify commit history.

Example:

```bash
git rebase -i HEAD~3
```

Common operations include:

* Reordering commits
* Squashing multiple commits
* Editing commit messages
* Removing unnecessary commits

Interactive rebasing is frequently used to **clean up commits before opening a pull request**.

---

## Best Practices

* Use rebase primarily for **local branch management**
* Avoid rebasing commits that have already been pushed
* Pull frequently to reduce conflicts
* Prefer rebase when preparing feature branches for merging

## Summary

* `git rebase` reapplies commits onto a new base commit

* It produces a **clean, linear project history**

* Rebase rewrites commits and changes commit hashes

* It is most useful for **local branch maintenance**

* Avoid rebasing shared or public branch history
