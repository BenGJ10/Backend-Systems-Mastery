# Git Initial Commands and Workflow

## Overview

When starting a project with Git, there is a **standard initialization and workflow** that converts a normal directory into a version-controlled repository.  
Understanding this initial setup is critical because every professional backend project follows the same foundational Git lifecycle, regardless of scale.

This section covers the **essential commands** and the **end-to-end flow** from project creation to the first commit.

---

## Initializing a Git Repository

### `git init`

Initializes a new Git repository in the current directory.

```bash
git init
```

What this command does:

* Creates a hidden `.git` directory
* Sets up internal metadata for version tracking
* Marks the directory as a Git repository

From this point onward, Git can track changes inside the folder.

---

### Understanding the `.git` Directory

The `.git` directory contains:

* Commit history
* Branch information
* Configuration settings
* References (HEAD, branches, tags)

Deleting this directory removes all version history.

---

## Checking Repository Status

### `git status`

Displays the current state of the repository.

```bash
git status
```

It shows:

* Untracked files
* Modified files
* Staged files
* Current branch

This command should be used **frequently** to understand what Git is tracking at any moment.

---

## Initial Workflow: From File to Commit

### Step 1: Create or Modify Files

Files are created or edited in the **working directory**.

Example:

```bash
touch App.java
```

At this stage, Git sees the file but does not track it.

### Step 2: Stage Files

Use `git add` to move files to the **staging area**.

```bash
git add App.java
```

Only staged files are included in the next commit.

### Step 3: Commit Changes

Create a commit to permanently record the staged changes.

```bash
git commit -m "Initial project setup"
```

This commit becomes part of the repository’s history.

---

## Typical Git Workflow (Conceptual)

```
Working Directory
       ↓
   git add
       ↓
  Staging Area
       ↓
  git commit
       ↓
 Repository History
```

This workflow ensures that changes are deliberate, reviewed, and well-documented.

## Viewing Commit History

### `git log`

Displays the commit history.

```bash
git log
```

Useful for:

* Reviewing project evolution
* Identifying changes
* Debugging regressions

---

### Compact History View

```bash
git log --oneline
```

Shows a concise summary of commits.

---

## Setting Up a Remote Repository (Basic)

### Adding a Remote

Links a local repository to a remote repository (e.g., GitHub).

```bash
git remote add origin https://github.com/user/repo.git
```

This allows pushing and pulling code between local and remote repositories.

### Verifying Remote Configuration

```bash
git remote -v
```

Shows configured remote URLs.

---

## First Push Workflow

### Pushing Code to Remote

```bash
git push -u origin main
```

What this does:

* Pushes local commits to the remote repository
* Sets `main` as the default upstream branch
