# Adding and Committing in Git

## Overview

In Git, **adding** and **committing** are the core operations that convert file changes into a permanent, versioned record.  

This process ensures that only intentional and reviewed changes become part of the project’s history, providing structure, traceability, and reliability to the development workflow.

Understanding this flow is essential because **every meaningful change in a backend system eventually becomes a commit**.

---

## The Git Workflow Context

Before diving into commands, it is important to understand Git’s three-stage model:

1. **Working Directory** – where files are created or modified
2. **Staging Area (Index)** – where selected changes are prepared for commit
3. **Repository (History)** – where committed changes are permanently stored

The `add` and `commit` commands move changes through these stages in a controlled manner.

---

## Adding Changes (`git add`)

### What Does `git add` Do?

`git add` moves selected changes from the **working directory** to the **staging area**.  
This step allows developers to explicitly choose which changes should be included in the next commit, rather than committing everything blindly.

Git does not automatically track file changes—**only staged changes are committed**.

---

### Common `git add` Commands

#### Add a Specific File
```bash
git add UserService.java
```

Stages only the specified file.

---

#### Add Multiple Files

```bash
git add UserService.java AuthController.java
```

Stages multiple selected files.

---

#### Add All Modified and New Files

```bash
git add .
```

Stages all changes in the current directory and its subdirectories.
This should be used carefully to avoid committing unintended changes.

---

### Why the Staging Area Matters

The staging area allows:

* Partial commits (only relevant changes)
* Clean and focused commit history
* Better code reviews and debugging

In backend projects, this is especially useful when:

* Code changes and configuration changes happen together
* Multiple fixes are made but should be committed separately

---

## Committing Changes (`git commit`)

### What Is a Commit?

A **commit** is a snapshot of the project at a specific point in time.
It permanently records:

* The exact changes staged
* The author of the change
* A descriptive message explaining *why* the change was made

Commits form a chronological history that represents the evolution of the system.

---

### Basic Commit Command

```bash
git commit -m "Add user authentication service"
```

This creates a commit with a concise message describing the change.


### Writing Good Commit Messages

A good commit message:

* Explains **what** changed and **why**
* Is concise but descriptive
* Represents a single logical change

Example:

```text
Fix race condition in payment processing
```

Avoid vague messages like:

```text
update
fix bug
changes made
```

---

### Commit Without Staging All Files

```bash
git commit -am "Update logging configuration"
```

This command:

* Automatically stages **modified tracked files**
* Does **not** include new untracked files

It is useful for quick fixes but should be used with caution.

---

## Viewing Commit History (`git log`)

### Basic Commit Log

```bash
git log
```

Displays:

* Commit hash
* Author
* Date
* Commit message

This helps developers trace when and why changes were introduced.

---

### Compact Log View

```bash
git log --oneline
```

Shows a concise, readable history, useful for quick inspection.

---

### Visual Branch History

```bash
git log --oneline --graph --all
```

This command provides a visual representation of:

* Branches
* Merges
* Commit relationships

Extremely useful in real-world backend repositories with multiple contributors.

---

## Inspecting Changes Before Committing

Before committing, it is good practice to review staged changes:

```bash
git status
```

Shows:

* Modified files
* Staged files
* Untracked files

```bash
git diff
```

Shows unstaged changes.

```bash
git diff --staged
```

Shows exactly what will be committed.

## Summary

* `git add` stages selected changes for commit

* The staging area allows controlled and meaningful versioning

* `git commit` creates a permanent snapshot of staged changes

* Commit messages are critical for code history clarity

* `git log` helps analyze and trace the evolution of the codebase
