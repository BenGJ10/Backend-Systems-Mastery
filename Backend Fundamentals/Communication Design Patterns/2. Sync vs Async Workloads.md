# Synchronous vs Asynchronous Workloads

## Overview

Backend systems process different kinds of workloads depending on **how and when** work is completed relative to a request.  
Two fundamental execution models are:

- **Synchronous workloads** — the caller waits for the operation to finish  
- **Asynchronous workloads** — the caller does not wait; work continues independently

Choosing between synchronous and asynchronous execution directly impacts **latency, scalability, resource utilization, reliability, and user experience**.

## Synchronous Workloads

### Definition

A synchronous workload is one where the **caller sends a request and waits for the result** before continuing execution.  
Control flow is blocked until the operation finishes and a response is received.

This style is common in traditional **request–response systems**.

---

### Characteristics of Synchronous Workloads

- Blocking execution
- Immediate result expected
- Strong temporal coupling between client and server
- Simpler programming model
- Direct error propagation

---

### Example: HTTP API Call

Client requests user details:

```text
Client → GET /users/42 → waits → receives response
```

Execution blocks until response arrives.

---

### When Synchronous Workloads Are Suitable

Use synchronous execution when:

* The user needs **instant feedback**
* Operations are **short-lived**
* The result determines the **next immediate step**
* Strong consistency is required

**Examples**

* Login authentication
* Fetching profile details
* Checking inventory before checkout
* Payment authorization request

---

## Asynchronous Workloads

### Definition

An asynchronous workload is one where the **caller initiates work and does not wait for completion**.
Work continues in the background, and the result may be delivered later through callbacks, messages, or polling.

---

### Characteristics of Asynchronous Workloads

* Non-blocking
* Decoupled in time
* Often message or event driven
* Better resource utilization
* Higher throughput under heavy load

---

### Example: Message Queue with Worker

```text
Client → submits job → returns immediately
Worker processes job later → result delivered asynchronously
```

Common tools: RabbitMQ, Kafka, SQS

---

### When Asynchronous Workloads Are Suitable

Use asynchronous execution when:

* Task is **long-running**
* Immediate response is not required
* Workload spikes must be buffered
* High scalability is needed
* Systems need resilience to downstream slowness

**Examples**

* Sending emails or notifications
* Video processing
* Report generation
* Background data aggregation
* Order fulfillment workflow

---

## Sync vs Async: Key Differences

| Aspect             | Synchronous               | Asynchronous                         |
| ------------------ | ------------------------- | ------------------------------------ |
| Caller waits       | Yes                       | No                                   |
| Time coupling      | Tight                     | Loose                                |
| Complexity         | Lower                     | Higher                               |
| Latency perception | Directly visible to user  | Often hidden                         |
| Scalability        | Limited by blocking calls | Higher through queues/workers        |
| Failure handling   | Immediate                 | Often via retries/dead-letter queues |

---

## Impact on Backend System Design

### Synchronous workload implications

* Higher request latency under load
* Thread blocking at the server
* Risk of cascading failures
* Requires strict timeout and retry policies

### Asynchronous workload implications

* Requires durable messaging infrastructure
* Harder observability and debugging
* Eventual consistency considerations
* Requires idempotency handling

---

## Combining Sync and Async in Real Systems

Most real systems use **both models together**.

Example: E-commerce order placement

1. User submits order → synchronous validation
2. Payment authorized → synchronous
3. Send confirmation email → asynchronous
4. Prepare shipment → asynchronous workflow

Good backend design is about **choosing the correct model per step**.

---

## Summary

* Synchronous workloads block until completion and are simpler but less scalable

* Asynchronous workloads continue in the background and improve scalability and resilience

* Sync is ideal for **short, user-facing, consistency-critical operations**

* Async is ideal for **long-running, background, or high-volume workloads**

* Real-world backend systems blend both models for optimal behavior
