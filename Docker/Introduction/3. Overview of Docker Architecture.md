# Overview of Docker Architecture

Docker architecture defines how Docker components interact to **build, manage, and run containers**.  

It follows a **client-server model**, where commands issued by the user are processed by the Docker engine to create and manage containers.

---

## High-Level Architecture

Docker consists of three primary components:

- **Docker Client**
- **Docker Daemon (Engine)**
- **Docker Registry**

These components communicate to handle container lifecycle operations.

```
Docker Client → Docker Daemon → Docker Registry
↓
Containers
```

---

## Docker Client

### What Is Docker Client?

The Docker client is the **interface through which users interact with Docker**.

Commonly used via the CLI:

```bash
docker build
docker pull
docker run
```

The client sends requests to the Docker daemon using a REST API.

---

### Responsibilities

* accepts user commands
* communicates with the daemon
* displays results
* manages build and run requests

The client itself does not create containers — it delegates execution.

---

## Docker Daemon (Docker Engine)

### What Is the Docker Daemon?

The Docker daemon (`dockerd`) is the **core background service** responsible for managing Docker objects.

It handles:

* building images
* running containers
* managing networks
* handling volumes
* enforcing isolation

The daemon listens for Docker API requests and executes them.

---

### Key Responsibilities

* container lifecycle management
* image creation and storage
* resource allocation
* networking configuration
* security isolation

Without the daemon running, Docker commands cannot function.

---

## Docker Registry

### What Is a Registry?

A Docker registry is a **central repository for storing and distributing Docker images**.

Examples:

* Docker Hub
* Amazon ECR
* Google Artifact Registry
* Private registries

When you run:

```bash
docker pull nginx
```

Docker downloads the image from a registry.

---

## How Components Work Together

### Example: Running a Container

Command:

```bash
docker run nginx
```

Flow:

1. Client sends request to daemon
2. Daemon checks for image locally
3. If missing, daemon pulls from registry
4. Daemon creates container
5. Container starts running

All heavy lifting is done by the daemon.

---

## Docker Objects in the Architecture

The daemon manages several objects:

### Images

Read-only templates used to create containers.

### Containers

Executable instances of images.

### Networks

Enable communication between containers.

### Volumes

Provide persistent storage outside containers.

---

## Local vs Remote Docker Engine

The Docker client can communicate with:

* a local daemon
* a remote Docker host

This enables managing cloud servers from a local machine.

Example use cases:

* remote deployments
* centralized container hosts
* CI/CD runners

---

## Why Docker Uses Client–Server Architecture

This design provides:

* separation of concerns
* remote management capability
* improved security boundaries
* scalable infrastructure control

It allows automation tools to interact directly with the daemon.

---

## Interview Questions

### 1. What are the main components of Docker architecture?

**Answer:**
Docker Client, Docker Daemon, and Docker Registry.

---

### 2. What does the Docker daemon do?

**Answer:**
It manages containers, images, networks, and executes Docker commands.

---

### 3. How does the Docker client communicate with the daemon?

**Answer:**
Through a REST API.

---

### 4. What is the role of a Docker registry?

**Answer:**
It stores and distributes Docker images.

---

### 5. Can the Docker client communicate with a remote daemon?

**Answer:**
Yes, enabling remote container management.

---

## Summary

* Docker uses a client-server architecture

* The client sends commands to the daemon

* The daemon builds images and runs containers

* Registries store and distribute images

* This architecture enables scalable and automated backend deployments
