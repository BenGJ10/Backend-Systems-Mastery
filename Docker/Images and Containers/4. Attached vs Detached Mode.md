# Attached vs Detached Containers

## Overview

When running a Docker container, you can choose whether it stays **connected to your terminal** or runs **independently in the background**.  
These two modes are known as:

- **Attached mode** — container runs in the foreground  
- **Detached mode** — container runs in the background  

---

## Attached Containers

### What Is Attached Mode?

In attached mode, the container is **linked directly to your terminal session**, allowing you to see logs and interact with the container in real time.

When the container stops, your terminal is released.

---

### How to Run a Container in Attached Mode

```bash
docker run nginx
```

Behavior:

* logs stream directly to your terminal
* stopping the container ends the session
* ideal for quick testing

---

### When to Use Attached Mode

Attached mode is useful when:

* debugging startup issues
* observing logs immediately
* testing container behavior
* running short-lived processes

It provides visibility but is not practical for production workloads.

---

## Detached Containers

### What Is Detached Mode?

Detached mode runs the container **in the background**, freeing your terminal while the container continues executing.

This is the standard mode for running backend services.

---

### How to Run a Container in Detached Mode

```bash
docker run -d nginx
```

The `-d` flag stands for **detached**.

Docker returns a container ID while the process continues running.

---

### Viewing Running Containers

```bash
docker ps
```

Shows active background containers.

---

### Viewing Logs from Detached Containers

```bash
docker logs <container-id>
```

Allows inspection without attaching.

---

### Stopping a Detached Container

```bash
docker stop <container-id>
```

Gracefully terminates the container.

---

## Switching Between Modes

### Attach to a Running Container

```bash
docker attach <container-id>
```

Reconnects your terminal to the container.

> Be cautious — exiting incorrectly may stop the container.

Safe detach shortcut:

```
CTRL + P + Q
```

---

## Attached vs Detached — Key Differences

| Feature            | Attached     | Detached            |
| ------------------ | ------------ | ------------------- |
| Terminal usage     | Occupied     | Free                |
| Runs in background | No           | Yes                 |
| Log visibility     | Immediate    | Via `docker logs`   |
| Ideal for          | Debugging    | Production services |
| Long-running apps  | Not suitable | Recommended         |

---

## Common Mistake

Running a backend server in attached mode and closing the terminal will stop the container.

Always use detached mode for persistent services.

---

## Interview Questions

### 1. What is an attached container?

**Answer:**
An attached container runs in the foreground and is directly connected to the terminal.

---

### 2. What is a detached container?

**Answer:**
A detached container runs in the background independently of the terminal.

---

### 3. Which flag is used to run a container in detached mode?

**Answer:**
The `-d` flag.

---

### 4. How do you view logs of a detached container?

**Answer:**
Using `docker logs <container-id>`.

---

### 5. Why is detached mode preferred for backend services?

**Answer:**
Because services must run continuously without occupying a terminal.

---

## Summary

* Attached containers run in the foreground with terminal interaction

* Detached containers run in the background

* Detached mode is standard for backend and production workloads

* Logs can be accessed without attaching

* Choosing the correct mode improves container management
