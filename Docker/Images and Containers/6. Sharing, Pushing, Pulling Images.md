# Sharing, Pushing, Pulling, and Using Docker Images

## Overview

Docker images become truly powerful when they can be **shared across machines and environments**.  

Registries such as Docker Hub allow developers to **store, distribute, and retrieve images**, enabling consistent deployments across teams and infrastructure.

The typical lifecycle involves:

- building an image  
- pushing it to a registry  
- pulling it on another system  
- running containers from it  

This workflow is foundational in modern backend and DevOps practices.

---

## What Is a Docker Registry?

A **Docker registry** is a centralized service that stores Docker images.

Common registries:

- Docker Hub (public default)
- Amazon ECR
- Google Artifact Registry
- Azure Container Registry
- Private self-hosted registries

Registries act as the **source of truth** for deployable artifacts.

---

## Sharing Docker Images

Images are shared by pushing them to a registry where others can pull them.

Before pushing, ensure the image is properly tagged:

```
username/backend-app:v1.0
```

The repository name must match your registry namespace.

---

## Step 1 — Authenticate with Registry

For Docker Hub:

```bash
docker login
```

You will be prompted for credentials or a personal access token.

Successful login allows image push operations.

---

## Step 2 — Push an Image

```bash
docker push username/backend-app:v1.0
```

Docker uploads image layers.

Key behavior:

* only missing layers are pushed
* existing layers are reused
* reduces upload time

---

## Step 3 — Pull an Image

To download an image from a registry:

```bash
docker pull username/backend-app:v1.0
```

If no tag is specified:

```bash
docker pull username/backend-app
```

Docker defaults to `latest`.

---

## What Happens During a Pull?

Docker:

1. checks local cache
2. downloads missing layers
3. reconstructs the image

This layered approach improves speed and bandwidth usage.

---

## Using Pulled Images

Once pulled, you can start containers immediately:

```bash
docker run username/backend-app:v1.0
```

Run in detached mode for services:

```bash
docker run -d username/backend-app:v1.0
```

Now the application is running inside a container.

---

## Keeping Images Updated

To fetch the newest version:

```bash
docker pull username/backend-app:latest
```

Then recreate containers using the updated image.

Containers do **not update automatically** when images change.

---

## Viewing Local Images

```bash
docker images
```

Useful for confirming successful pulls.

---

## Removing Unused Images

```bash
docker image prune
```

Removes dangling images and frees disk space.

---

## Common Mistakes

### Forgetting to Tag Correctly

Incorrect:

```
backend-app:v1
```

Correct:

```
username/backend-app:v1
```

Without namespace, push will fail.

---

### Assuming Containers Auto-Update

Pulling a new image does not affect running containers — they must be recreated.

---

### Using `latest` in Production

Always prefer explicit version tags for predictable deployments.

---

## Interview Questions

### 1. What is a Docker registry?

**Answer:**
A Docker registry is a centralized system for storing and distributing container images.

---

### 2. What command pushes an image to a registry?

**Answer:**
`docker push <image-name>`.

---

### 3. How do you download an image from Docker Hub?

**Answer:**
Using `docker pull <image-name>`.

---

### 4. Do running containers update automatically after pulling a new image?

**Answer:**
No, containers must be recreated to use the updated image.

---

### 5. Why are registries important in backend deployments?

**Answer:**
They provide a consistent and versioned source for deployable artifacts.

---

## Summary

* Docker registries store and distribute images

* Images are shared using `docker push`

* Images are retrieved using `docker pull`

* Containers run from pulled images

* Versioned images enable reliable backend deployments
